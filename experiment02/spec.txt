WebAssembly vs JavaScript — Performance Experiment (experiment02)
===================================================================

Purpose
-------
Real-time camera app that draws Canny edge outlines. Three implementations compute
the same visual effect; the app compares speed (FPS and compute milliseconds).

- JavaScript — Canny in plain JS
- WebAssembly — same algorithm in C++, full resolution
- Full Optimization — same Canny in WASM at half resolution, upscaled for display

All pages live in a flat `experiment02/` folder. Open via `file://` or a local server.


Hard constraints (must follow when rebuilding)
----------------------------------------------
1. No ES modules (`type="module"`). Use `<script src>` tags + small inline init
   blocks. ES modules fail on `file://` in this project.
2. WASM must be embedded in JS (`SINGLE_FILE=1`) — separate `.wasm` fetch fails
   on `file://`.
3. WASM heap is fixed at 64 MB (`ALLOW_MEMORY_GROWTH=0`). C++ must pre-allocate
   all workspace in `init_buffers`; no per-frame vector growth.
4. Do not log from C++ with `EM_ASM` + `UTF8ToString` — breaks the Emscripten build.
5. Canvas pixel size must track `video.videoWidth` / `video.videoHeight`, not CSS size.
6. After editing C++: run `./build-wasm.sh`, then hard-refresh WASM pages.

Canny algorithm (JS and C++ — independent implementations, identical parameters)
--------------------------------------------------------------------------------
Pipeline: RGBA → grayscale → Gaussian blur → Sobel → non-max suppression → hysteresis → RGBA edges (white on black).

Parameters (both `image-processing.js` and `image-processing.cpp`):
  CANNY_THRESHOLD1 = 50
  CANNY_THRESHOLD2 = 150
  CANNY_APERTURE_SIZE = 3
  CANNY_L2_GRADIENT = false   (L1 magnitude: |dX| + |dY|)

Fairness: every camera frame is processed when toggle is ON (no frame skipping).
JS and WASM options use full camera resolution. Full Optimization uses the same
Canny parameters at half resolution (`processScale` 0.5, clamped 0.25–1.0).

Metrics overlay (canvas top-left)
---------------------------------
Always present on test pages:
  #resolutionLabel     e.g. "1920 × 1080 px" or "1920 × 1080 px (proc 960×540)"
  #displayFps          "Display FPS: N" — passthrough rate
  #processingFps       "Processing FPS: N" — hidden until processing first enabled

JavaScript page only:
  #jsComputeMs         "JS compute (avg): X.XX ms" — hidden until processing ran

WASM pages only (wasm.html, full-optimization.html):
  #wasmTimingGroup     Container, hidden until processing ran
    #wasmCopyInMs      "WASM copy in (avg): X.XX ms"
    #wasmComputeMs     "WASM compute (avg): X.XX ms"
    #wasmTotalMs       "WASM total (avg): X.XX ms"

Rolling window: 1 second for all averages (FpsTracker / TimingTracker).

FPS rules:
  Display FPS     — updates while processing OFF; freezes when ON
  Processing FPS  — hidden until first ON; updates while ON; freezes when OFF

Timing rules:
  Sampled only while processing ON; overlay freezes when OFF (same as Processing FPS)
  JS compute      — `cannyEdgeDetect()` wall time only
  WASM copy in    — `HEAPU8.set` input
  WASM compute    — `_process_canny` only
  WASM total      — copy in + compute + copy out

localStorage (test-runner `persistOverlay` → `persistFps`):
  Keys:
    exp02-js-fps
    exp02-wasm-fps
    exp02-full-optimization-fps
  Value:
    { displayFps, processingFps, computeMs, updatedAt }
  computeMs — rolling 1 s average of computation only (JS `cannyEdgeDetect` or WASM
  `_process_canny`). `TimingTracker.getComputeMs()` returns the average, or the last
  single-frame measurement if the window is empty. `persistFps` keeps the previous
  computeMs when no new sample is available.
  Saved on: every overlay update, processing toggle OFF, pagehide, and visibility hidden.

Shared module: test-runner.js
-------------------------------
Exp02TestRunner.startTestRunner({
  title,                  // sets #pageTitle
  storageKey,             // localStorage key
  processFrame,           // (imageData, w, h) => ImageData
  onReady, onResize,      // optional; arity 2 or 4 in full optimization
  onError,                // optional camera failure
  optimization,           // 'none' (default) or 'full'
  processScale,           // default 0.5
})

Camera: auto-start on load. Fallback constraints:
  1080p rear → 720p rear → default rear → any camera
Waits up to 10 s for video dimensions.

Toggle #toggleProcessing switches processingActive; green `.active` class when ON.

Persistence helpers in test-runner:
  readComputeMs(now)  — Exp02Js/Exp02Wasm getComputeMs, then frozen overlay fallback
  persistOverlay(now) — reads computeMs, calls persistFps with last known value
  pagehide / visibilitychange:hidden — final persist before leaving page

WebAssembly build (build-wasm.sh)
---------------------------------
Requires Emscripten (`em++`).

  em++ image-processing.cpp -O3 \
    -s WASM=1 -s SINGLE_FILE=1 \
    -s MODULARIZE=1 -s EXPORT_NAME='ImageProcessingModule' \
    -s EXPORTED_FUNCTIONS='["_init_buffers","_get_input_buffer","_get_output_buffer","_process_canny","_malloc","_free"]' \
    -s EXPORTED_RUNTIME_METHODS='["HEAPU8"]' \
    -s ALLOW_MEMORY_GROWTH=0 -s INITIAL_MEMORY=67108864 \
    -o image-processing-wasm.js

Prepend build stamp to output:
  globalThis.EXP02_WASM_BUILD_ID="<YYYYMMDDHHMMSS>";

C++ exports (image-processing.cpp):
  init_buffers(width, height)   — allocate input/output RGBA + workspace vectors
  get_input_buffer()            — uint8_t* RGBA in
  get_output_buffer()           — uint8_t* RGBA out
  process_canny(width, height)  — run pipeline in-place

Operational notes
-----------------
- Grant camera permission when prompted.
- Summary updates after visiting test pages; values stored in localStorage.
- Compute time (avg) on summary requires processing ON for at least a few frames.
- Hard-refresh test pages after code changes (`?v=6` or bump N).
- WASM load confirmed when step log shows: WebAssembly runtime initialized (build …)
- Use Copy log on test pages to export debug traces.
- If WASM behaves oddly after C++ edits: rebuild, hard-refresh, check EXP02_WASM_BUILD_ID.
